# Plan: Bucket 3 - Light Agent Platform Contract
_Locked via grill-with-docs (re-shaped: mechanical branches decided by Claude with stated defaults; Greg decided the two judgment branches). Terms per CONTEXT.md. Extracted against Scout (Bucket 1) AND Validation (Bucket 2), per locked decision: Validation is NOT a listener._

## Goal

Make the kl-agents platform spine REAL instead of convention: extract the shared agent infrastructure (DB client / config seam, run lifecycle, structured logging, config surface) into `shared/` so it does not depend on any single agent, and write a thin "what makes a kl-agent" contract doc that the next agent (Validation) is built on. (The fetcher base was a candidate but is DEFERRED - see the mechanical-branches note; moving it would re-introduce a `shared -> agents` import.) Light contract: a small, documented, enforced surface - NOT a base class or framework. The deliverable proves itself by Validation building cleanly on it with zero copy-paste of Scout internals and zero `shared/ -> agents/` imports.

## Grill decisions (what was settled, and by whom)

**Q1 - Contract scope (Claude, Greg confirmed the governing rule):** Doc + minimal extraction. **"Minimal" = a code move is in scope ONLY if it (a) unblocks Validation or (b) fixes a real seam.** No refactor-for-elegance, no renames, no speculative abstraction. Every move below is justified against this test.

**Decision A - agent_runs over-fit (Greg):** Core columns + JSONB `metrics`. The shared `agent_runs` table currently hardcodes four LISTENER counts (`posts_fetched`, `posts_dedup_skipped`, `posts_classified`, `posts_queued`, all NOT NULL DEFAULT 0). Validation fetches/classifies nothing, so those would be permanent zeros - a meaningful-looking lie. Fix: the contract owns the UNIVERSAL columns; each agent owns its own work-specific metrics in a JSONB column. No fake zeros, no per-agent column sprawl.

**Decision B - scope horizon (Greg):** Generalize where free, no speculative code. Extract against Scout+Validation as locked; where a choice is equally easy either way, pick the one that obviously generalizes (e.g. JSONB metrics). Write ZERO code for hypothetical agent #4 (prospecting, new sources). Generality is a tiebreaker, never added surface. (NOTE: the fetcher base was the apparent "free generality" win, but live-code review showed it is NOT free - it imports `agents.scout.models` - so it is deferred, not extracted. Decision B does not override the Q1 "no new `shared -> agents` violation" floor.)

## Key decisions & tradeoffs

- **Why extract code at all (not doc-only):** `shared/db/client.py:16` imports `agents.scout.config` - the shared foundation depends on the agent built on it. A doc cannot fix a Python import. Validation would inherit the tangle. (Q1.)
- **Why not a base class:** three agents do not justify an inheritance framework; CONTEXT.md locked "NOT a heavy framework." Convention + a thin enforced invariant is lighter and sufficient.
- **Why JSONB metrics over fixed columns:** non-listener agents make the listener counts meaningless; JSONB lets each agent record true metrics without schema churn or fake zeros. (Decision A.)
- **Why move logging:** it is structurally agent-agnostic, only Scout-located, and Validation needs identical structured logging - a real shared seam. The move makes agent-name injection explicit at the formatter (round-2 finding #2), not new surface.
- **Why DEFER the fetcher base (not move it):** it looked like free generality but is not - `fetchers/base.py:16` imports `agents.scout.models`, so moving it to `shared/` would re-introduce a `shared -> agents` violation (or force the models to move, scope Validation does not need). It moves with the shared fetch-model extraction when a 3rd source lands. (Round-1 finding #2.)
- **No ADR proposed:** these are reversible, unsurprising extractions of an existing pattern; they fail the "hard to reverse + surprising + genuine trade-off" ADR test. The contract doc + this plan are the record.

## Risks / open questions

- **R1 - Scout regression during extraction.** Scout is live (YouTube path). Every slice keeps Scout running and green; 3a-3b are behavior-preserving moves (same logic, new location). Mitigation: tests green at every checkpoint; mypy clean; a real Scout run before declaring done.
- **R2 - migration 0010 on a live table.** `agent_runs` has real Scout history. Plan: additive `metrics` column + relax (not drop) the count columns + **backfill metrics from existing counts** (finding #6) so no historical totals are lost. Migration runs BEFORE any metrics-writing code (finding #1). No data loss; old rows stay valid. Physically dropping the deprecated columns + removing the transition legacy-write is a LATER bucket once Echo + dashboards are audited (finding #5).
- **R3 - fetcher-base move: RESOLVED as DEFERRED (finding #2).** It is not free - `base.py` imports `agents.scout.models`, so the move would either re-introduce a `shared -> agents` violation or pull the models too (scope Validation doesn't need). Dropped from Bucket 3; moves with the shared fetch-model extraction when a 3rd source lands. Blocks nothing.
- **R4 - partial-failure semantics for a non-listener.** Scout's model is success/partial/failed (partial = a source failed). Validation has no sources to partially fail. RESOLVED: the contract states `partial` is OPTIONAL - agents that cannot partially fail simply never emit it (success/failed only). The DB status comment is rewritten agent-neutral in 3b (finding #8).

## Out of scope

- Any change to Validation's logic (Bucket 2); Bucket 3 only provides the spine Validation will build on.

- A base class, plugin system, or agent registry.
- Physically dropping the deprecated count columns AND removing the transition legacy-column mirror-write (a later bucket, after Echo cutover + dashboard audit - findings #5/#6).
- Moving the fetcher base / shared fetch models to `shared/` (deferred to the 3rd-source slice - finding #2).
- Any code for agent #4 / prospecting / new fetcher sources (Decision B: zero speculative code).
